Overview

Learn why we created a style guide and why you should adhere to it.

Why do we need a style guide?

Docs created a style guide for a few reasons. The main reason is because Docs is a template. Templates have specific ways they need to be filled out otherwise they might break. For this, we have set some standards that we think are in the best interest for the end user reading your documentation, while also providing the least amount of code to make it have semantic meaning.

Another reason is simply because there is no standard other than what you use to document your project. If you're using Docs for that, then this is the standard you will need to follow. Of course, you could break free from this and do whatever you like, but it may require more work for you in the end. That is simply the price of lots of customizations.

What do I need to do?

Simply use this page for reference when you are creating docs. Look at the the overall structure of the sections, subsections, API items, etc. while you are creating your docs. They should all flow naturally without confusing the end user. If you had to read something multiple times to understand it, chances are your users will too, so rework it.

You should also take a look at the source of this page if you ever wonder how to create elements like we have. This page isn't meant as a template to start from, but it does contain templates for all of the common elements you will use when creating with Docs.

Elements

Common elements used when creating docs

When writing documentation you end up repeating a lot of basic structure, elements, and such so you can have consitency. We have created majority of the basic elements and styling so you can have documentation done fast. Check out what is available to you.

Note: In addition to the stylings below, you can use styles that are packaged with the Skeleton CSS framework. This would include items such as columns, list formats, buttons, etc.

Basic

Here are some basic elements which apply to the whole page. By basic elements, we mean normal semantic markup that you would see in every HTML page. These tags do not need any specific classes or IDs unless noted.

Paragraphs

<p>Content goes here</p>
Example

This is a sample paragraph. No classes, no IDs. You can add bold or italic text in here as you please.

Lists: Bullets, Numerical, Etc.

<!-- unformatted -->
<ul>
  <li>Unformatted list item #1</li>
  <li>Unformatted list item #2</li>
</ul>

  <li>Formatted list item #1</li>

Notes, Importants

Sometimes you need to point something out to your users. This may be a word of caution or a "Hey, seriously!" item. For this we have notes and importants. The text inside of the elements can be anything. It does not need to say "Note" or "Important".

Notes

<p class="note"><strong>Note:</strong> You should probably read this.</p>
Example

Note: You should probably read this.

Important

<p class="important"><strong>Important:</strong> You really need to read this.</p>
Example

Important: You really need to read this.

<p class="important"><strong>Caution:</strong> Be careful when using this function!</p>
Example

Caution: Be careful when using this function!

Code Blocks

If you are writing documentation for code, you will need lots of sample code. For this we have included Highlighter.js to provide syntax coloring for code blocks. This script will automatically run. By default, the script will assume your code is HTML and format it for so. If you want to use a specific language, you can add a class to it to tell the script to format for that language instead. See how to do this with the examples below.

Note: We only load the HTML, CSS, and JavaScript languages by default. If you want to change which ones to load, set this in your config options.

Code

<pre><code><p>Basic HTML code</p>

<p>And a second paragraph</p></code></pre>
Example 
<pre><code><p>Basic HTML code</p>

<p>And a second paragraph</p></code></pre>
<!-- code with javascript formatting -->
<pre><code class="javascript">// open a new window
function openMe() {
  window.open();
}</code></pre>
Example 
// open a new window
function openMe() {
  window.open();
}

Examples, Output Samples

If you want to show the sample output of something you would use our Examples or Output box. We made this to ensure the user does not get confused with what is sample output and what is describing the actual documentation. This box will visually surround the output for easy reading. The example box is not required when posting code samples, but you can include them if you'd like. These are required when you want to show sample HTML output.

Example

The first element inside the example is the caption. This can be set to whatever you like. Default is Example. However, if the words Output or Sample suit your preview better, use one of those.

<section class="example">
	<small class="caption">Example</small>
		Example Content
	</section>
</section>
